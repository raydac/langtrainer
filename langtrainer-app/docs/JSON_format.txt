JSON file format describes language learning resources in the DialogDefinition shape (menuName, description, languages, lines, optional path, optional modules, optional inputEqu, optional shuffled, optional per-line images).

Bundled files are normally registered in common/jsons/index.json (see index_json.txt in this folder). The virtual keyboard is configured separately (keyboard.txt).

    {
        "menuName" :  "Menu name of the dialog",
        "path" : "Optional menu path for resource lists (e.g. travel/russian-english/50 words). Segments separated by / or \\ only; spaces allowed inside a segment. Omitted or at root when absent.",
        "modules" : [ "DIALOG", "!IMAGES" ],
        "description" : "Description of the dialog shown as tooltip",
        "langA" : human-readable name of language A as string,
        "langB" : human-readable name of language B as string,
        "rtl" : [ "Hebrew" ],
        "lines" : [
            {
                "A" : text line in language A,
                "B" : text line in language B,
                "image" : optional Base64-encoded image for this line
            }
            ...
            {
                "A" : text line in language A,
                "B" : text line in language B
            }
        ]
    }

    Optional field "modules" is an array of module id strings (LangTrainerModuleId names: DIALOG,
    FLY_GAME, CROSSWORD, BRICKS, BARS, IMAGES). If omitted or empty, every module may list the
    resource. A plain module id is an allow rule: if at least one plain id is present, only those
    modules list the resource. A module id prefixed with "!" is a deny rule: that module must not
    list the resource even when the resource is otherwise allowed. Deny rules are supported by
    module resource lists, but the Editor module exposes only plain allow checkboxes and does not
    edit "!" entries.

    Examples:
       "modules" : [ "DIALOG", "FLY_GAME" ]   // only Dialog and Words list it
       "modules" : [ "!IMAGES" ]               // every module except Images lists it
       "modules" : [ "DIALOG", "!IMAGES" ]     // Dialog lists it; Images is explicitly excluded

    Optional field "rtl" is an array of human-readable language names that should be treated as
    right-to-left. Values are compared case-insensitively with "langA" and "langB". If the field is
    omitted, empty, or does not contain the selected language, modules keep the default
    left-to-right behavior.

    Example:
       "langA" : "English",
       "langB" : "Hebrew",
       "rtl" : [ "Hebrew" ]

    Optional field "image" on a line object stores one embedded picture for that line as a Base64
    text string (no data: prefix required; a data:image/...;base64,... prefix is also accepted).
    Allowed formats after decoding: PNG, JPG/JPEG, SVG. Maximum decoded size: 64 KB. SVG must not
    contain scripts, DTDs, or external entities; rasterization is capped at 1024 px per side. Omitted when the
    line has no picture. Current training modules ignore line images unless they declare image
    support; invalid or oversized images block activation only for those modules. The Images module
    uses only lines that have both embedded images and non-empty text.

    Example:
       "lines" : [
         { "A" : "apple", "B" : "яблоко", "image" : "<Base64 PNG bytes>" }
       ]

    There can be special field "inputEqu" in the JSON file, it contains array of objects and its example below

     "inputEqu": [
        {
          "key": [
            "е",
            "Е"
          ],
          "value": [
            "ё",
            "Ё"
          ]
        },
        {
          "key": [
            "ё",
            "Ё"
          ],
          "value": [
            "е",
            "Е"
          ]
        },
     ]

    The array "key" contains all expected chars entered by user to be substituted by expected char from the "value" section.

    The field describes rules of substitution for enter user chars based on expected next char. For instance if in entering work
    expected Russian char 'Ё' but user presses Russian 'Е' or 'е' then expected 'Ё' will replace entered е. "Key" chars expected
    as input and "Value" chars allowed for substitution. So if expected 'Ё' and user presses 'е' then there will be substitution by 'Ё'.

    It is possible to have more variants as substitution for pressed keys, for instance in Estonian language

      ],

        "Ё"
      ]
    },
    {
      "key": [
        "a",
        "A"
      ],
      "value": [
        "ä",
        "Ä"
      ]
    },
    {
      "key": [
        "o",
        "O"
      ],
      "value": [
        "ö",
        "Ö",
        "õ",
        "Õ"
      ]
    },
    {
      "key": [
        "u",
        "U"
      ],
      "value": [
        "ü",
        "Ü"
      ]
    },
    {
      "key": [
        "s",
        "S"

      "value": [

The root of the JSON can contain boolean flag "shuffled" which shows that by default data lines can be shuffled by modules which allow shuffling (for instance Dialog).

Example:
   "shuffled" : true

By default such missing flag recognized as false.
